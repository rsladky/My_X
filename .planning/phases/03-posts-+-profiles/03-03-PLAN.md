ProtectedParentRoute pattern (from RESEARCH.md Pattern 1):
```rust
// In app.rs Routes block — auth_user is RwSignal<Option<AuthUser>>
<ProtectedParentRoute
    path=StaticSegment("")
    view=AuthenticatedLayout
    condition=move || Some(auth_user.get().is_some())
    redirect_path=|| "/login"
>
    <Route path=StaticSegment("") view=HomePage/>
    <Route path=path!("/:username") view=ProfilePage/>
</ProtectedParentRoute>
```

Note: `condition` returns `Option<bool>`. `Some(false)` → redirect. `Some(true)` → render layout.
Using `Some(auth_user.get().is_some())` means unauthenticated users (signal = None) get `Some(false)` and redirect to /login. There is a brief window on page load before JWT validates, but the research OPEN QUESTION analysis concluded this is acceptable for Phase 3 (user logs back in and JWT validates in < 200ms).

Dynamic route parameter (from RESEARCH.md Pattern 2):
```rust
use leptos_router::hooks::use_params_map;
let params = use_params_map();
let username = move || params.read().get("username").unwrap_or_default();
```

Pitfall: /login and /register MUST be siblings of ProtectedParentRoute, NOT children of it (would cause infinite redirect loop).
